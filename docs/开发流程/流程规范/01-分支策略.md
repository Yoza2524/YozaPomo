# Git 分支策略

> 本文档定义 YozaPomo 项目的 Git 分支管理规范。

---

## 1. 分支模型

采用简化的 Git Flow 模型，适合个人项目：

```
main ─────────────────────────────────────────────────>
  │              │              │              │
  ├─ feat/xxx ───┤              │              │
  │              ├─ feat/yyy ───┤              │
  │              │              ├─ fix/zzz ────┤
  │              │              │              │
  └──────────────┴──────────────┴──────────────┘
```

## 2. 分支类型

| 分支 | 命名格式 | 用途 | 生命周期 |
|------|----------|------|----------|
| 主分支 | `main` | 稳定版本，可随时发布 | 永久 |
| 功能分支 | `feat/功能名` | 开发新功能 | 开发完成后合并 |
| 修复分支 | `fix/问题描述` | 修复 Bug | 修复后合并 |
| 文档分支 | `docs/内容` | 更新文档 | 更新后合并 |
| 预发布 | `release/v版本号` | 发布前准备 | 发布后删除 |

## 3. 分支操作流程

### 3.1 开发新功能

```bash
# 1. 从 main 创建功能分支
git checkout main
git pull origin main
git checkout -b feat/floating-todo-tags

# 2. 开发并提交
git add .
git commit -m "feat(floating): 添加 TODO 标签基础组件"

# 3. 开发完成后合并到 main
git checkout main
git merge feat/floating-todo-tags

# 4. 推送到远程
git push origin main

# 5. 删除功能分支（可选）
git branch -d feat/floating-todo-tags
```

### 3.2 修复 Bug

```bash
# 1. 从 main 创建修复分支
git checkout main
git checkout -b fix/timer-pause-issue

# 2. 修复并提交
git commit -m "fix(timer): 修复暂停后恢复倒计时不准的问题"

# 3. 合并到 main
git checkout main
git merge fix/timer-pause-issue
git push origin main
```

### 3.3 同步远程

```bash
# 拉取最新代码
git pull origin main

# 如果有冲突，解决后
git add .
git commit -m "merge: 解决合并冲突"
```

## 4. Commit Message 规范

### 格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 示例

```
feat(floating): 实现悬浮窗透明背景

- 配置 decorations: false 和 transparent: true
- 实现 macOS 和 Windows 的透明效果
- 添加背景毛玻璃 CSS

Closes #5
```

```
fix(db): 修复数据库迁移失败问题

- 修复迁移版本号判断逻辑
- 添加数据库文件存在性检查

Fixes #12
```

## 5. 标签管理

### 版本号规则

采用语义化版本（SemVer）：

```
v<major>.<minor>.<patch>

示例：
v0.1.0  - 首次发布
v0.1.1  - Bug 修复
v0.2.0  - 新功能
v1.0.0  - 正式版
```

### 创建标签

```bash
# 打标签
git tag -a v0.1.0 -m "首次发布：基础悬浮窗功能"

# 推送标签
git push origin v0.1.0
```

## 6. 个人项目简化规则

由于是个人项目，采用以下简化规则：

1. **不需要 Pull Request**：直接在本地合并
2. **不需要 Code Review**：使用 Claude Code 辅助审查
3. **main 分支始终可用**：合并前确保代码可运行
4. **功能分支可选**：简单修改可直接在 main 上修改
5. **提交粒度适中**：一个功能一个提交，不要太大也不要太小
